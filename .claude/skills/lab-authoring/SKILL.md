# Lab Authoring (실습 노트 작성)

학습자가 따라 치면 통합 프로젝트(원격탐사 변화탐지 앱)가 한 단계씩 완성되는 실습 노트를 쓴다. 교재의 "실습 위주" 핵심이다.

## 작성 원칙

- **복붙 가능성이 생명**. 추상 설명이 아니라 실제로 실행되는 코드/명령. 환경 설정(계정·인증·설치)을 건너뛰지 않는다.
- **한 단계 = 한 가지 동작**. 작게 쪼개고, 각 단계 끝에 검증 지점("이게 보이면 성공")을 둔다.
- **이어지는 프로젝트**. 각 실습은 이전 실습 산출물을 입력으로 받아 다음 결과를 만든다. 입력·출력 파일/변수명을 명시해 다음 실습과 끊기지 않게 한다.
- **실제 동작 코드만**. API를 추측하지 말고, 참고 프로젝트 `C:\dev\dkchangedetection`·`C:\dev\dkidi`의 검증된 코드 패턴을 근거로 삼는다.
- **개념은 링크로**. 설명은 최소화하고 개념 노트로 위키링크. 실습은 "하는 법"에 집중.

## 실습 노트 템플릿

```markdown
---
tags: [주제/gee, 유형/실습, 난이도/입문]
---

# {순번}. {이 실습에서 만드는 것}

> [!info] 이 실습을 마치면
> {완성되는 산출물 한 줄}. 약 {N}분.
> **선행**: [[이전 실습]] 의 산출물 `{파일/변수}` 가 필요하다.
> **개념**: [[관련 개념]]

## 준비
{필요한 계정·설치·인증. 이미 했으면 건너뛰어도 된다고 명시}

## 단계 1 — {동작}
{무엇을 왜 하는지 한 줄}

\`\`\`python
# 복붙 가능한 실제 코드
\`\`\`

> [!success] 확인
> {이게 보이면 성공: 출력 예시/스크린샷 자리표시자 ![[assets/...png]]}

## 단계 2 — ...

## 막히면
> [!warning] 자주 나는 오류
> - 증상 → 원인 → 해결

## 다음
이 실습의 산출물 `{파일/변수}` 은 다음 실습 [[다음 실습]] 의 입력이 된다.
```

## 주제별 유의점

- **GEE**: Earth Engine 인증/초기화를 첫 실습에서 확실히. Code Editor(JS)와 Python API 중 교재가 택한 방식을 일관 사용. AOI·기간·밴드를 변수로 빼서 학습자가 바꿔보게 한다.
- **GeoAI**: 변화탐지/분류의 입력(두 시점 영상)과 출력(변화 마스크/분류맵)을 명확한 산출물로. 결과 해석 단계를 포함.
- **Claude Code**: 바이브 코딩 흐름 — 자연어로 요구→생성→실행→수정. 실제 프롬프트 예시와 기대 동작을 보여준다. 참고: `https://github.com/taehojo/vibecoding`.
- **Harness**: 에이전트 팀으로 분석 자동화. 학습자가 쓰는 도구 자체를 다루는 메타 단계이므로, 작은 예부터 점진적으로.

## 흔한 함정

- 환경 설정 생략 → 초급자는 첫 줄에서 막힌다. 인증·설치를 빠뜨리지 않는다.
- 검증 지점 없는 긴 코드 → 어디서 틀렸는지 모른다. 단계마다 "확인"을 둔다.
- 단계 간 입출력 불일치 → tech-validator가 잡는 1순위 버그. 산출물 이름을 다음 실습과 맞춘다.
- 동작 안 하는 코드 → 참고 프로젝트로 대조하고, 불확실하면 tech-validator에 검증 요청.

## 체크리스트

- [ ] 모든 코드가 복붙 실행 가능한가 (환경 설정 포함)
- [ ] 단계마다 검증 지점이 있는가
- [ ] 선행 산출물·후행 입력이 명시돼 끊기지 않는가
- [ ] 코드가 참고 프로젝트 패턴에 근거하는가
- [ ] 개념은 링크로 넘기고 "하는 법"에 집중했는가
