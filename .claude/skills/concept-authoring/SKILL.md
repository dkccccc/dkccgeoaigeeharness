# Concept Authoring (개념 노트 작성)

초급 원격탐사 연구자가 실습을 이해하는 데 필요한 개념을, 통합 프로젝트(변화탐지)의 맥락에서 짧고 명료하게 설명하는 Obsidian 노트를 쓴다.

## 작성 원칙

- **왜 → 무엇 → 어떻게의 입구**. 정의로 시작하지 말고, 학습자가 마주칠 문제 상황에서 출발한다. "두 시점 영상을 그냥 빼면 구름 때문에 엉뚱한 변화가 잡힌다 → 그래서 정사보정/대기보정이 필요하다".
- **한 노트 = 한 개념**. 길어지면 분할하고 위키링크로 잇는다.
- **비유 우선**. 추상 개념은 원격탐사 연구자에게 익숙한 일상/현장 비유로 먼저 직관을 준 뒤 정확한 정의를 붙인다.
- **용어는 친절하게**. 전문 용어 첫 등장 시 한 줄 정의 + 그 용어의 개념 노트로 위키링크. "JSON", "assertion" 같은 코딩 용어도 설명 없이 쓰지 않는다.
- **실습으로 연결**. 노트 끝에 "이 개념을 쓰는 실습: [[lab-...]]"을 둬서 개념이 어디서 손에 잡히는지 보여준다.
- **코드·명령은 쓰지 않는다**. 실습 단계는 lab-builder의 몫. 개념 노트는 이해를 만든다.

## 노트 템플릿

```markdown
---
tags: [주제/geoai, 유형/개념, 난이도/입문]
---

# {개념명}

> [!question] 왜 필요한가
> {학습자가 마주칠 문제 상황 한두 문장}

## 한마디로
{비유 + 한 줄 정의}

## 조금 더
{초급자가 알아야 할 만큼만. 수식·세부는 최소화하거나 접어둔다}

> [!tip] 우리 프로젝트에서는
> {변화탐지 앱의 어느 단계에서 이 개념이 왜 등장하는지}

## 관련
- 함께 보면 좋은 개념: [[...]]
- 이 개념을 쓰는 실습: [[lab-...]]
```

## Obsidian 표현 활용

- **콜아웃**으로 시선을 잡는다: `> [!question]`(문제 제기), `> [!tip]`(프로젝트 연결), `> [!warning]`(흔한 오해), `> [!note]`(보충).
- **위키링크** `[[노트명]]`은 매니페스트의 정확한 파일명(확장자 제외)을 쓴다. 표시 텍스트가 다르면 `[[파일명|보이는 글자]]`.
- **프론트매터 태그**는 매니페스트/볼트 규약의 태그 체계를 따른다.

## 흔한 함정

- 백과사전식 나열 — 초급자에게 필요 없는 깊이는 과감히 버린다.
- 깨진 위키링크 — 거는 링크의 파일명을 매니페스트와 대조하고, 실습 노트명은 lab-builder와 맞춘다.
- 개념과 실습의 중복 서술 — 개념은 "왜", 실습은 "어떻게". 겹치면 위키링크로 넘긴다.

## 체크리스트

- [ ] 문제 상황(왜)에서 시작하는가
- [ ] 전문 용어에 비유/정의/링크가 붙는가
- [ ] 통합 프로젝트 맥락("우리 프로젝트에서는")이 있는가
- [ ] 거는 위키링크가 매니페스트 파일명과 일치하는가
- [ ] 한 노트가 한 개념에 집중하는가
