# 📎 프롬프트 모음

이 교재에서 Claude Code에게 건네는 모든 프롬프트를 한곳에 모았습니다. **복붙해서 바로 쓰세요.** 그대로 따라 해도 되고, `<영역>`·`<기간>` 같은 부분을 여러분 주제에 맞게 바꿔도 됩니다.

## 0장 · 첫 바이브 코딩


## 1장 · 말하는 대로 — 위성영상 뷰어

**1-1. 뼈대 만들기 (사이드바 + 지도)**
```
나는 원격탐사 연구자인데 코딩은 초보야. 0장에서 만든 datawa_study00/app.py(위성영상
한 장을 띄우는 Streamlit 웹앱)를 이어서 키우고 싶어. datawa_study01/app.py로 새로
만들어줘. 먼저 영상 기능은 빼고 웹앱 골격부터 만들어줘. st.sidebar에 중심 위도·경도·
반경(km)을 입력받는 칸과 [분석 / 보기] 버튼을 두고, 본문에는 geemap.foliumap으로 빈
지도를 m.to_streamlit()으로 띄워줘. ee.Initialize()는 0장처럼 try/except로 감싸고,
주석은 한국어로 달아줘.
```

**1-2. 기능 붙이기 (고른 영역 영상 + 구름 필터)**
```
좋아, 골격이 떴어. 이제 [분석 / 보기] 버튼을 누르면 사이드바에서 고른 영역의 위성영상이
지도에 보이게 해줘. 위도·경도·반경으로 관심 영역(ROI)을 만드는 make_roi(lon, lat,
radius_km) 함수를 추가해줘 — 점을 반경만큼 buffer한 뒤 그 사각형(bounds)을 쓰면 돼.
그리고 그 영역의 COPERNICUS/S2_SR_HARMONIZED 영상 중 구름이 많은 건 자동으로 걸러서
가장 맑은 한 장을 골라 자연색(B4,B3,B2)으로 보여줘. 변화를 보려는데 구름이 방해돼서
걸러야 하거든.
```

**1-3. 점검하기 (구름 필터 전후 비교)**
```
방금 넣은 구름 필터가 실제로 동작하는지 눈으로 확인하고 싶어. 같은 영역·기간에서 가장
흐린 영상과 가장 맑은 영상을 둘 다 지도에 올려 켜고 끄며 비교할 수 있게, 임시 점검용
코드를 알려줘. 확인이 끝나면 어떻게 다시 지우는지도 알려줘.
```

## 2장 · NDVI 식생지도 앱

**2-1. 뼈대 만들기 (NDVI 계산 추가)**
```
나는 원격탐사 연구자이고 코딩은 초보야. 1장에서 만든 app.py(사이드바로 지역 골라
위성영상 보는 Streamlit 뷰어)가 잘 돌아가. 같은 파일에 한 가지만 추가하자. 지금
만들어진 관심 영역(roi)에 대해 NDVI(식생지수)를 계산하는 ndvi_for_roi(roi, start, end)
함수를 만들어줘. 구름 적은 가장 맑은 Sentinel-2 한 장에서 계산하고, 밴드 이름이 "NDVI"인
이미지 하나를 반환만 하면 돼. 아직 지도에 그리거나 색칠하진 말고, 함수가 깔끔히
이미지를 반환하는 것부터.
```

**2-2. 기능 붙이기 (색지도 + 범례)**
```
좋아. 이제 사이드바에 "보기 모드" 라디오 버튼을 추가해줘. "자연색"이면 지금처럼
위성영상 그대로 보여주고, "NDVI"를 고르면 방금 만든 ndvi_for_roi 로 NDVI를 구해 지도에
색으로 보여줘. 식생이 적은 곳은 갈색, 많은 곳은 초록색으로 칠하고, 색이 무슨 값을
뜻하는지 알 수 있게 범례(colorbar)도 같이 띄워줘.
```

**2-3. 점검하기 (물 vs 숲)**
```
NDVI가 제대로 계산된 건지 확인하고 싶어. 사이드바 좌표를 물(강·호수) 위로 옮겼을 때와
숲 위로 옮겼을 때, 영역의 NDVI 평균값을 화면에 같이 보여줘. 물이 낮고 숲이 높게 나오면
제대로 된 거지?
```

## 3장 · 변화를 탐지하기 — 미니 변화탐지 앱

**3-1. 뼈대 만들기 (두 시점 NDVI 차분)**
```
좋아, 2장에서 만든 app.py 가 잘 돌아가. 자연색이랑 NDVI를 보여주는 그 Streamlit 앱이야.
여기에 변화탐지를 더하자. 같은 app.py 안에, 그 앱의 ndvi_for_roi 함수를 그대로 써서 같은
영역의 두 시점(이전 기간 / 이후 기간) NDVI를 각각 구한 다음, 둘을 빼서 변화량 이미지를
반환하는 함수 ndvi_change(roi, before, after) 를 만들어줘. 새 영상을 다시 받지 말고 기존
ndvi_for_roi 를 두 번 호출했으면 좋겠어. 반환 이미지의 밴드 이름은 "NDVI_change", 값은
"이후 − 이전"으로. 디자인·화면은 아직 신경 쓰지 마.
```

**3-2. 기능 붙이기 (변화 히트맵 + 범례)**
```
이제 그 변화량을 앱 화면에서 보여주자. 사이드바에 이전 기간(시작·끝)과 이후 기간
(시작·끝)을 입력하는 칸을 추가하고, 보기 모드에 "변화"를 추가해줘(자연색 / NDVI / 변화).
"변화"를 고르면 ndvi_change 결과를 지도에 발산형 히트맵으로 칠해줘 — 식생이 줄어든 곳은
빨강, 변화 없는 곳은 흰색, 늘어난 곳은 초록. 0을 가운데 흰색에 두고 양끝을 대칭으로
(−0.3 ~ +0.3) 잡고, 색이 무슨 값인지 알 수 있게 범례(colorbar)도 같이 띄워줘.
```

**3-3. 점검하기 (아는 변화지역으로 검증)**
```
변화가 진짜인지 화면에서 검증하고 싶어. "변화" 모드일 때, 지도 위에 요약 통계를 같이
보여줘 — 영역 안에서 변화량의 절댓값이 임계값(예: 0.2)보다 큰 픽셀 중 큰 감소(빨강)와
큰 증가(초록)가 각각 몇 %인지 st.metric 카드로. 내가 사이드바 좌표를 개발지나 벌채지로
옮기면 감소 비율이 높게 나와야 맞는 거지?
```

## 4장 · 쓸 만한 변화탐지 서비스

**4-1. 뼈대 만들기 (구름 임계값 + 요약 통계)**
```
지금 3장 변화탐지 앱(app.py)이 잘 돌아가. 이걸 '쓸 만한 서비스'로 키우고 싶어.
1) 변화탐지를 run_change_detection(roi, before, after, cloud_pct) 함수 하나로 감싸줘.
   변화 이미지 하나만 주지 말고, 평균 변화량·식생 감소 면적 비율·증가 면적 비율·영역
   넓이(km²)를 같이 담은 딕셔너리로 돌려줘.
2) 사이드바에 구름 임계값 슬라이더(cloud_pct, 5~60%)를 추가해서, 그 값으로 영상을 거르게 해줘.
3) [분석 / 보기]를 누르면 그 요약 통계를 st.metric 카드 4개(평균 변화량·감소%·증가%·면적)로
   화면에 띄워줘. 맑은 영상이 없으면 친절히 안내하고 멈추게 해줘.
```

**4-2. 기능 붙이기 (GeoJSON/CSV 다운로드)**
```
이제 이 결과를 파일로 내려받게 하고 싶어. 영역을 사이드바에서 고른 격자 크기(예: 2km)로
나눠서 각 칸의 평균 변화량을 계산하고, 칸별 (중심좌표, 변화량) 표를 만들어줘. 그 표를
화면에 pandas로 미리 보여주고, 아래에 CSV 내려받기 / GeoJSON 내려받기 두 개의
st.download_button 을 달아줘. CSV는 엑셀로, GeoJSON은 QGIS로 바로 열리게 하고, 파일
이름엔 기간이 들어가게 자동으로 지어줘.
```

**4-3. 점검하기 (내려받은 파일 값 확인)**
```
방금 내려받은 CSV를 엑셀로 열었어. 값이 제대로 들어갔는지 같이 확인하고 싶어. 셀이 몇
개고, 변화량(cell_change)의 최소·최대·평균이 얼마면 정상인지 알려줘. 너무 크면(±1 이상)
뭘 의심해야 해? 그리고 화면 미리보기 표와 내려받은 파일의 값이 같은지 어떻게 대조하면 돼?
```

## 5장 · AI 에이전트 팀 — 여러 지역 자동 분석

**5-1. 뼈대 만들기 (여러 지역 모드 + 반복)**
```
지금 app.py 는 한 지역만 분석해. 이걸 그대로 두고, 앱 맨 위에 "한 지역 / 여러 지역" 모드
선택(st.radio)을 추가해줘. "여러 지역" 모드에서는 텍스트 영역에 이름,위도,경도 를 한 줄씩
붙여넣게 하고, 그 목록을 받아 각 지역에 4장 run_change_detection 을 차례로 호출하는
batch_change_detection(aoi_list, before, after, cloud_pct) 함수를 만들어줘. 한 지역에서
맑은 영상이 없어 실패해도 멈추지 말고, 그 지역만 건너뛰고 다음으로 넘어가게 해줘. 진행
상황은 화면에 진행 막대로 보여줘.
```

**5-2. 기능 붙이기 (결과 표 + 보고서)**
```
batch_change_detection이 모은 결과를 한 표로 묶고 싶어. 지역 이름, 평균 변화량, 식생
감소 %, 식생 증가 %를 칸으로 한 표를 st.dataframe 으로 화면에 띄우는 build_report(results)
를 만들어줘. 변화가 가장 많이 감소한 지역이 표 맨 위로 오게 정렬하고, 표 아래에 "가장
주의해서 볼 지역" 한 줄 요약과 표를 CSV로 내려받는 버튼도 붙여줘. 분석에 실패해 건너뛴
지역은 표 아래 펼침에 이유와 함께 보여줘.
```

**5-3. 점검하기 (표 값 교차확인)**
```
방금 만든 표의 값들이 상식과 맞는지 화면에서 점검하고 싶어. 평균 변화량이 보통 어느
범위면 정상이고 너무 크면 뭘 의심해야 하는지, 각 지역의 감소%와 증가% 합이 말이 되는지
표 위에 짚어줘. 값이 이상한 지역이 있으면 그 행을 눈에 띄게 표시(경고)해주고, "가장
주의해서 볼 지역은 한 지역 모드로 다시 띄워 지도로 확인하라"는 안내도 붙여줘.
```

## 6장 · 세상에 내보내기 — 배포와 공유

**6-1. 뼈대 만들기 (배포 준비: requirements · 서비스계정)**
```
좋아, 5장 앱(한 지역 + 여러 지역 분석)이 로컬에서 잘 돌아가. 이제 이걸 배포할 수 있게
준비하고 싶어. 두 가지를 해줘. 첫째, 필요한 패키지를 담은 requirements.txt를 만들어줘
(streamlit, geemap, earthengine-api). 둘째, Earth Engine 인증을 클라우드에서도 되게
고쳐줘 — st.secrets에 서비스 계정 JSON이 있으면 ee.ServiceAccountCredentials로 초기화하고,
없으면 로컬처럼 ee.Initialize()를 쓰도록 try/except로 분기해줘. 분석 로직
(run_change_detection·batch_change_detection)은 그대로 두고, 왜 이렇게 갈리는지 한국어
주석으로 설명해줘.
```

**6-2. 기능 붙이기 (Streamlit Cloud 배포 → 링크 공유)**
```
이제 이 앱을 무료로 배포해서 링크로 공유하고 싶어. Streamlit Community Cloud에 올리는
절차를 처음부터 단계별로 알려줘. GitHub 저장소에 뭘 올려야 하는지(app.py + requirements.txt),
share.streamlit.io에서 어떻게 연결하는지, 그리고 6-1에서 만든 서비스 계정 인증이
동작하도록 Google Cloud에서 서비스 계정 키를 발급해 Earth Engine에 등록하고 그 JSON을
Streamlit Secrets에 [gee_service_account]로 넣는 방법까지 짚어줘.
```

**6-3. 점검하기 (MCP 맛보기 + 배포 확인)**
```
MCP(Model Context Protocol)가 뭔지 초보자 눈높이로 짧게 설명해줘. 그리고 내 변화탐지 결과
파일들(여러 지역 분석에서 내려받은 보고서)이 든 폴더를 Claude Code가 직접 읽을 수 있게,
가장 간단한 MCP 서버(예: filesystem) 하나를 연결하는 방법을 알려줘. 설정 파일에 뭘 적어야
하는지 한 줄씩 짚어주고, 연결이 됐는지 확인하는 법도 알려줘.
```

## 프롬프트 잘 쓰는 4가지 습관

1. **나를 소개한다** — "나는 원격탐사 연구자, 코딩 초보야" → Claude가 눈높이를 맞춘다
2. **한 번에 하나씩** — 뼈대 먼저, 기능은 그다음. 한꺼번에 시키면 어디가 틀렸는지 모른다
3. **왜를 말한다** — "구름을 걸러줘" 보다 "변화를 보려는데 구름이 방해돼서 걸러줘"
4. **결과를 의심한다** — "정말 되는지 확인하는 법도 알려줘"
